硅步千里，分布式理论加强内功修炼

本章节目标

  • 了解大型互联网系统演进
  • 掌握架构设计思想
  • 掌握CAP理论
  • 掌握BASE理论

  • 中国互联网发展历程
    • 1994 - 2000年:从四大门户到搜索
    • 2001-2009年:从搜索到社交化网络
    • 2010- 至今:移动互联网和自媒体

单体应用架构到分布式架构的演进

  • 单体应用架构
    • Linux+Apache+MySQL+PHP
    • 网站刚起步的时候，流量少。用一台服务器就可以满足
  • 应用服务与数据服务分离
    • 访问增加，带来更大压力
    • 增加配置解决临时问题
    • 从横向进行扩展，一分为多
  • 不同服务器配置要求不同
服务器类型 不同配置要求
应用服务器 CPU配置高
数据库服务器 磁盘IO快和稳，内存足够大
文件服务器 磁盘足够大

  • 开始引入缓存

    • 大量交易访问数据库压力增大，响应延迟
    • 给数据库减负，把请求在前面就处理完
    • 在数据库前面加入缓存

  • 应用服务器集群部署

    • 所有的请求直接打到负载均衡组件
    • 单台服务器 >多台服务器
    • 软件: Apache / Nginx / HaProxy / LVS
    • 硬件:F5

  • 数据库的读写分离

    • 并非所有请求都操作缓存，且存在缓存失效场景，对数据库访问造成压力
    • 利用数据库主从复制机制，将读和写操作分开，进一步提升性能

  • 数据库访问模块

    • 对接多个数据源、处理读写分离、甚至分库分表等
    • 独立于应用程序存在的
    • Apache ShardingSphere

  • 内容分发网络 - CDN

  • CDN服务进行资源加速

    • 把静态资源提前缓存到各地的边缘服务器
    • 可以有效地降低DDoS攻击

  • 全面分布式化

    • 随着规模不断发展产生数据越来越多，单文件件服务器、单数据库服务器也渐渐地达到瓶颈。利用集群化横向扩展解决问题
    • HDFS、TFS、FastDFS
    • NoSQL & 搜索引擎
      • Lucene是一个搜索引擎的开发工具包:全文检索
      • Solr和Elasticsearch它们是基于Lucene开发的
      • 常见的NoSQL有Mongodb和ELasticsearch

  • 单体应用面临诸多问题和挑战

    • 单体应用的代码膨胀，研发成本变高，难实现敏捷交付
    • 代码维护成本变高，开发人员交接困难
    • 测试成本变高，回归的工作量变大，给测试带来巨大工作量
    • 可扩展性差，技术升级时需考虑整体，无法单独调整

  • “大而全的单体”拆分为多个“独立的应用”

  • 应用拆分和解耦

    • 对应用模块进行拆分之后，通过消息中间件进行交互
    • 此外还有异步处理应用解耦、流量削峰等作用

  • 常见的消息中间件

消息中间件特性
ActiveMQ 历史悠久，使用Java语言编写的，功能全面，基本上MQ的功能它都有，因此也比较复杂。
RabbitMQ 目前主流之一，跟Spring框架师出同门，因此Spring对RabbitMQ的支持最好
Kafka 高性能方面比较有优势。但一开始是基于大吞吐量数据场景设计，在消息可靠性方面比较弱
RocketMQ 它相对前面几个，有后发优势，结合了各方优点综合能力比较好。
  • 模块拆分引发的问题
    • 代码冗余，多处相同的代码维护起来困难
    • 复杂度扩散
    • 共用一套数据库资源

引入服务化架构

SOA与微服务对比


SOA 微服务
共同点 都是在分布式环境下，以服务为中心的松耦合应用架构模式服务之间是通过定义明确的接口和协议来进行相互通信 都是在分布式环境下，以服务为中心的松耦合应用架构模式服务之间是通过定义明确的接口和协议来进行相互通信
目标 解决的是异构系统的服务化 解决互联网系统快速交付
规模 相对粗粒度 细粒度
适用性 跨企业、大企业跨部门的复杂应用系统建设 适合小型团队的持续化发布
